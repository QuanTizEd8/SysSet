```
│       ├── validate.yml
│       └── release.yaml

```

The key design principle is **separation of concerns**:

| Layer | Directory | Purpose |
|---|---|---|
| Feature source | `src/` | What is packaged and published to GHCR |
| Tests | `test/` | Verification — never shipped |
| Shared library | `lib/` | Canonical bash helpers — never shipped directly |
| Build tooling | `sync-lib.sh`, `bootstrap.sh` | Distributes shared code into `src/` |
| Dev environment | `.devcontainer/` | Container config for local development |

## Feature source — `src/`

Each subdirectory of `src/` is one devcontainer feature. The directory name
is the feature's ID and must match the `"id"` field in its
`devcontainer-feature.json`.

### Feature directory anatomy

```
src/<feature>/
├── devcontainer-feature.json   ← Feature metadata, options, and dependsOn
├── install.sh                  ← Generated bootstrap (git-ignored, synced from bootstrap.sh)
├── scripts/
│   ├── install.sh              ← The real installer script (bash ≥4)
│   ├── _lib/                   ← Synced copy of lib/ (git-ignored)
│   │   ├── logging.sh
│   │   ├── os.sh
│   │   ├── net.sh
│   │   ├── ospkg.sh
│   │   ├── shell.sh
│   │   └── git.sh
│   └── <helper>.sh             ← Optional helper scripts sourced by install.sh
├── dependencies/
│   └── base.txt                ← OS packages required before install.sh runs
└── files/                      ← Optional static files deployed by the feature
```

The devcontainer CLI calls `install.sh` (the bootstrap) at image build time.
The bootstrap ensures `bash` is available, then execs
`scripts/install.sh "$@"`, which is the real installer written in bash ≥4.
`scripts/install.sh` sources helpers from `scripts/_lib/` — the local copy
of the shared library.

The `install.sh` at the feature root and the entire `scripts/_lib/` directory
are **generated artefacts** — they are git-ignored and regenerated by
`sync-lib.sh` (see below). Never edit them directly.

## Tests — `test/`

Every feature that has scenarios has a matching subdirectory under `test/`.
See [Testing](testing.md) for the full guide.

`test/run-fail-scenarios.sh` is a test runner for "expected-to-fail"
scenarios — cases where the feature should exit non-zero (e.g. invalid
inputs). It is invoked by CI after the main scenario tests.

## Shared library — `lib/`

`lib/` contains the canonical source for the shared bash helper library.
There is one file per concern:

| File | Namespace | Provides |
|---|---|---|
| `os.sh` | `os::` | `os::require_root` |
| `logging.sh` | `logging::` | `logging::setup`, `logging::cleanup` |
| `net.sh` | `net::` | URL fetching with retry, curl/wget detection, CA cert installation |
| `ospkg.sh` | `ospkg::` | Cross-distro package manager abstraction, manifest parsing and execution |
| `shell.sh` | `shell::` | Shell config path detection, theme/plugin resolution |
| `git.sh` | `git::` | Idempotent shallow cloning |

Each file uses a guard variable (`_LIB_<NAME>_LOADED`) to prevent
double-sourcing when multiple scripts source the same library.

**Edit only the files in `lib/`** — never the copies under
`src/*/scripts/_lib/`. After editing, run `bash sync-lib.sh` to propagate
changes (or simply stage the file; the pre-commit hook does it automatically).

## Build tooling — `sync-lib.sh` and `bootstrap.sh`

### The bootstrap pattern

The devcontainer specification guarantees only POSIX `sh` is available when
`install.sh` runs. Bash 4+ features used by the library are therefore not
available at that point. The bootstrap pattern resolves this:

- `bootstrap.sh` is a POSIX `sh` script. Its only job is to ensure `bash` is
  present and then `exec bash scripts/install.sh "$@"`.
- `scripts/install.sh` is written in bash ≥4 and uses all library features
  freely.

`bootstrap.sh` knows how to install bash on every supported distro (APT, APK,
DNF, microdnf, YUM, Zypper, Pacman). It is the single source of truth — one
copy is distributed to every feature root as `install.sh`.

### The sync mechanism

`sync-lib.sh` automates the distribution of shared files:

```
bootstrap.sh  →  src/*/install.sh           (one copy per feature)
lib/          →  src/*/scripts/_lib/        (one copy per feature)
```

It auto-discovers features by finding every `devcontainer-feature.json` under
`src/` that also has a `scripts/` subdirectory. No list is hard-coded.

```bash
# Sync all features
bash sync-lib.sh

# Verify copies are up to date (exits non-zero if stale)
bash sync-lib.sh --check
```

CI runs `bash sync-lib.sh` as the first step of every workflow job to ensure
the working tree is consistent before building or testing.

### Pre-commit hook

[Lefthook](https://github.com/evilmartians/lefthook) is configured to run
`bash sync-lib.sh` automatically whenever `lib/*` or `bootstrap.sh` is staged
for a commit, so the generated copies in `src/` are always in sync with the
canonical sources.

```yaml
# lefthook.yml
pre-commit:
  commands:
    sync-lib:
      glob: "{lib/*,bootstrap.sh}"
      run: bash sync-lib.sh
```

Install the hooks once after cloning:

```bash
npx lefthook install
# or: brew install lefthook && lefthook install
```

## Generated / git-ignored copies

`.gitignore` excludes all generated artefacts:

```
# Generated copies — synced by sync-lib.sh.
# Never commit these; CI runs sync-lib.sh after checkout.
src/**/_lib/
src/*/install.sh
```

Reasons for not committing:
- They are exact copies — committing them would create merge conflicts and
  make diffs noisy every time a library file is touched.
- CI regenerates them from the canonical sources on every run, so the working
  tree is always correct.
- The git-ignore makes it immediately apparent when someone accidentally edits
  a generated copy (the change disappears on the next sync).

## Development environment — `.devcontainer/`

`.devcontainer/` contains configuration for the **repository's own dev
container** — the container you use when working on this repo. It does not
contain feature source code.

```
.devcontainer/
├── devcontainer.json           ← Main dev container (Node.js + Docker-in-Docker)
├── _src                       → ../src  (symlink — see below)
├── install-shell/
│   └── devcontainer.json       ← Dev container for testing install-shell locally
├── install-miniforge/
│   └── devcontainer.json
└── install-podman/
    └── devcontainer.json
```

The main `devcontainer.json` uses a Node.js base image (for the devcontainer
CLI) with Docker-in-Docker (for running container builds during testing), and
installs the devcontainer CLI via `updateContentCommand`.

### Per-feature dev containers

The `install-shell/`, `install-miniforge/`, and `install-podman/`
subdirectories each contain a `devcontainer.json` that references the local
feature via a relative path. These exist so you can open a VS Code window
scoped to a specific feature's dev container — useful for exercising the
feature interactively during development.

### The `_src` symlink

The devcontainer CLI enforces a constraint: locally-referenced features must
reside **inside** the `.devcontainer/` directory (it validates paths using
`path.relative('.devcontainer', child)` and rejects any path containing
`..`). Since `src/` lives at the repo root (not inside `.devcontainer/`), a
symlink is used to satisfy this constraint:

```
.devcontainer/_src  →  ../src
```

The symlink's apparent path (`.devcontainer/_src/install-shell`) passes the
CLI's check. At build time Node.js follows the symlink and reads the real
files from `src/`.

Per-feature `devcontainer.json` files reference features using this path:

```jsonc
{
  "features": {
    "../_src/install-shell": {}
  }
}
```

The `_` prefix signals that `_src` is infrastructure, not a real source
directory.

## CI workflows — `.github/workflows/`

| File | Trigger | Purpose |
|---|---|---|
| `test.yaml` | push, PR, manual | Discover changed features, run scenario tests and fail-scenario tests |
| `validate.yml` | PR, manual | Validate all `devcontainer-feature.json` files via `devcontainers/action` |
| `release.yaml` | manual only | Publish features to GHCR and generate documentation PRs |

All three workflows run `bash sync-lib.sh` as their first step before
building or testing, ensuring that `_lib/` copies and `install.sh` stubs are
always present even though they are git-ignored.

See [Testing](testing.md) for how the test workflow discovers and runs
scenarios. See [Publishing](publishing.md) for the release workflow.
